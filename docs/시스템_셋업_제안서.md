## 2. 문제점 및 원인

### 2.1 생산수불 문제

| 문제 유형 | 원인 | 영향도 | 우선순위 |
|---------|------|--------|---------|
| **수식 불일치** | 수동 입력 오류, 시스템 로직 미검증 | 🔴 HIGH | 1 |
| **월간 연결성 오류** | 전월 마감 미완료, BOH 수동 조정 | 🔴 HIGH | 2 |
| **LOTRUN 불일치** | 집계 로직 오류, 원천 데이터 누락 | 🔶 MEDIUM | 3 |
| **음수 재고** | 출고 과다, 투입 미반영 | 🔴 HIGH | 1 |
| **원천 미반영** | I/F 타이밍, 배치 실패 | 🔶 MEDIUM | 4 |

### 2.2 재고수불 문제

| 문제 유형 | 원인 | 영향도 | 우선순위 |
|---------|------|--------|---------|
| **수식 불일치** | 입출고 누락, 수동 조정 미반영 | 🔴 HIGH | 1 |
| **생산-재고 불일치** | 타이밍 차이, 구분 코드 불일치 | 🔴 HIGH | 2 |
| **수량/금액 불일치** | 단가 계산 오류, 반올림 차이 | 🔶 MEDIUM | 3 |
| **월간 연결성 오류** | 기초재고 수동 조정 | 🔴 HIGH | 2 |
| **음수 재고** | 출하 과다, 생산 입고 누락 | 🔴 HIGH | 1 |

### 2.3 판매수불 문제

| 문제 유형 | 원인 | 영향도 | 우선순위 |
|---------|------|--------|---------|
| **재고-판매 불일치** | 거래명세서 vs 실재고 차이 | 🔴 HIGH | 1 |
| **매출 누락** | ERP 연계 오류, 수동 입력 누락 | 🔴 HIGH | 2 |
| **환율 차이** | 환산 시점 불일치 | ⚠️ LOW | 5 |

## 3. 시스템 셋업 로드맵

### 3.1 Phase 1: 데이터 정리 (4주)

```
Week 1-2: 현황 진단
├─ 진단 SQL 실행 및 리포트 생성
├─ 오류 유형별 분류 (ERROR/WARNING/INFO)
├─ 우선순위 결정 (High/Medium/Low)
└─ 해결 계획 수립

Week 3-4: 데이터 클렌징
├─ 음수 재고 정리 (물리적 실사 필요)
├─ 누락 데이터 복구 (원천 시스템 확인)
├─ 수식 오류 수정 (BOH/IN/OUT/EOH 재계산)
└─ 월간 연결성 확보 (전월 EOH = 당월 BOH)
```

**주요 작업:**
1. **음수 재고 해결**
   - 물리적 실사 실시
   - 시스템 재고 vs 실물 재고 차이 조정
   - 조정 사유 문서화

2. **원천 데이터 복구**
   - DW_PROD_FRONT/BACK 테이블 확인
   - 누락된 Lot/Run 데이터 추가 입력
   - 생산수불 재집계

3. **생산-재고 연결**
   - 구분 코드 통일 (제품/반제품)
   - 타이밍 규칙 정의 (당일/익일 반영)
   - 자동 검증 규칙 추가

### 4.2 검증 규칙 정의

#### 4.2.1 생산수불 검증 규칙

| 규칙코드 | 규칙명 | SQL 로직 | 심각도 | 실행주기 |
|---------|-------|---------|--------|---------|
| **PS001** | 생산수불 수식 검증 | `EOH = BOH + IN + BONUS - OUT - LOSS - NG` | ERROR | 매일 |
| **PS002** | SUBUL/LOTRUN 일치 | `SUM(DOI_PROD_SUBUL) = SUM(DOI_PROD_LOTRUN)` | ERROR | 매일 |
| **PS003** | 월간 연결성 | `전월 EOH = 당월 BOH` | ERROR | 월초 |
| **PS004** | 원천 데이터 누락 | `DW_PROD_* NOT IN DOI_PROD_LOTRUN` | WARNING | 매일 |
| **PS005** | 음수 재고 | `BOH < 0 OR EOH < 0` | ERROR | 매일 |

#### 4.2.2 재고수불 검증 규칙

| 규칙코드 | 규칙명 | SQL 로직 | 심각도 | 실행주기 |
|---------|-------|---------|--------|---------|
| **ST001** | 재고 수식 검증 | `EOH = BOH + INPUT - OUT` | ERROR | 매일 |
| **ST002** | 생산→재고 연결 | `PROD.OUT_MONTH = STOCK.INPUT_PROD` ⭐ | ERROR | 매일 |
| **ST003** | 월간 연결성 | `전월 EOH = 당월 BOH` | ERROR | 월초 |
| **ST004** | 수량/금액 일치 | `STOCK.EOH = STOCK_COST.EOH_QTY` | WARNING | 매일 |
| **ST005** | 음수 재고 | `BOH < 0 OR EOH < 0` | ERROR | 매일 |

#### 4.2.3 판매수불 검증 규칙 (향후 추가)

| 규칙코드 | 규칙명 | SQL 로직 | 심각도 | 실행주기 |

## 5. 운영 프로세스

### 5.1 월마감 프로세스 (운영 중심)

```
익월 1일
├─ 전월 생산/입고/출하 데이터 입력 완료 확인

├─ 1단계: 데이터 정합성 검증 실행
│   ├─ 생산수불 검증 (PS001~PS005)

│   └─ 재검증 실행

익월 3일
├─ 3단계: 최종 검증 (ERROR 0건 확인)
├─ 월마감 승인 (책임자)
└─ 원가계산 시작 가능

익월 5일
└─ 원가계산 완료 (C0003000)
    └─ 재료비 배부 → 가공비 배부 → 단가 계산
```

```java
@Service
public class ValidationServiceImpl implements ValidationService {
    
    @Autowired
    private List<BaseValidator> validators;
    
    @Override
    @Transactional
    public ValidationResult executeValidation(ValidationRequest request) {
        // 1. 로그 생성
        Long logId = createValidationLog(request);
        
        // 2. 규칙별 검증 실행
        List<ValidationError> errors = new ArrayList<>();
        for (BaseValidator validator : validators) {
            if (validator.isApplicable(request)) {
                errors.addAll(validator.validate(request));
            }
        }
        
        // 3. 결과 저장
        saveValidationErrors(logId, errors);
        
        // 4. 로그 업데이트
        updateValidationLog(logId, errors.size());
        
        return new ValidationResult(logId, errors);
    }
}
```
